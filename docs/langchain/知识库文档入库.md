# 知识库文档入库——文本分割、向量化与MD5去重

知识库文档入库是RAG系统的数据基础建设环节。一篇文档从上传到可检索，需要经过内容去重、文本分割、向量化和存储四个步骤。其中，MD5去重避免重复文档浪费存储空间，文本分割将长文档切分为语义合理的片段，向量化将文本转为可计算相似度的向量表示，最终存入Chroma向量数据库。本文将详细介绍`KnowledgeBaseService`的设计与实现。

## 核心概念

- **KnowledgeBaseService**：知识库管理的核心服务类，封装了文本分割、向量化和存储的完整流程。
- **MD5去重**：使用`hashlib`计算文本内容的MD5哈希值，与已存储的哈希值比对，相同则跳过入库，避免重复数据污染知识库。
- **RecursiveCharacterTextSplitter**：LangChain提供的递归字符文本分割器，按分隔符优先级列表递归尝试分割，确保在语义边界处切分。
- **Chroma.add_texts**：Chroma向量数据库的文本添加方法，自动调用嵌入模型将文本向量化后存储，同时支持附加元数据（metadata）。
- **upload_by_str**：文档入库的主入口方法，接收文本字符串和文件名，执行去重→分割→向量化的完整流程。

## 代码示例

```python
import os
import hashlib
from langchain_chroma import Chroma
from langchain_ollama import OllamaEmbeddings
from langchain_text_splitters import RecursiveCharacterTextSplitter
from datetime import datetime

class KnowledgeBaseService(object):
    def __init__(self):
        self.chroma = Chroma(
            collection_name=config.collection_name,
            embedding_function=OllamaEmbeddings(model=config.embedding_model_name),
            persist_directory=config.persist_directory,
        )
        self.spliter = RecursiveCharacterTextSplitter(
            chunk_size=config.chunk_size,
            chunk_overlap=config.chunk_overlap,
            separators=config.separators,
            length_function=len,
        )

    def upload_by_str(self, data: str, filename):
        md5_hex = get_string_md5(data)
        if check_md5(md5_hex):
            return "[跳过]内容已经存在知识库中"
        if len(data) > config.max_split_char_number:
            knowledge_chunks = self.spliter.split_text(data)
        else:
            knowledge_chunks = [data]
        metadata = {
            "source": filename,
            "create_time": datetime.now().strftime("%Y-%m-%d %H:%M:%S"),
            "operator": "小曹",
        }
        self.chroma.add_texts(knowledge_chunks, metadatas=[metadata for _ in knowledge_chunks])
        save_md5(md5_hex)
        return "[成功]内容已经成功载入向量库"
```

## 代码解析

**1. 初始化Chroma向量数据库**

```python
self.chroma = Chroma(
    collection_name=config.collection_name,
    embedding_function=OllamaEmbeddings(model=config.embedding_model_name),
    persist_directory=config.persist_directory,
)
```
创建Chroma实例时指定三个关键参数：
- `collection_name`：集合名称，同一集合内的向量共享索引空间。
- `embedding_function`：嵌入函数，`OllamaEmbeddings`调用本地Ollama服务将文本转为向量，`add_texts`时自动调用。
- `persist_directory`：持久化目录，向量数据自动保存到磁盘。

**2. 初始化文本分割器**

```python
self.spliter = RecursiveCharacterTextSplitter(
    chunk_size=config.chunk_size,
    chunk_overlap=config.chunk_overlap,
    separators=config.separators,
    length_function=len,
)
```
- `chunk_size`：每个文本块的最大字符数。
- `chunk_overlap`：相邻块的重叠字符数，保证语义连贯性。
- `separators`：分隔符优先级列表，递归尝试从高优先级到低优先级分割。
- `length_function`：长度计算函数，`len`按字符数计算。

**3. MD5去重检查**

```python
md5_hex = get_string_md5(data)
if check_md5(md5_hex):
    return "[跳过]内容已经存在知识库中"
```
`get_string_md5`计算文本内容的MD5哈希值，`check_md5`检查该哈希是否已存在于记录文件中。如果已存在，说明相同内容的文档已经入库，直接跳过避免重复。

**4. 条件分割**

```python
if len(data) > config.max_split_char_number:
    knowledge_chunks = self.spliter.split_text(data)
else:
    knowledge_chunks = [data]
```
只有当文本长度超过`max_split_char_number`阈值时才进行分割，短文本直接作为单个块存储，避免不必要的分割开销。

**5. 元数据附加与入库**

```python
metadata = {
    "source": filename,
    "create_time": datetime.now().strftime("%Y-%m-%d %H:%M:%S"),
    "operator": "小曹",
}
self.chroma.add_texts(knowledge_chunks, metadatas=[metadata for _ in knowledge_chunks])
save_md5(md5_hex)
```
- 每个文本块附加相同的元数据，记录来源文件名、入库时间和操作者。
- `add_texts`将文本块列表和对应的元数据列表一起存入Chroma，嵌入模型自动将文本向量化。
- 入库成功后保存MD5哈希值，标记该文档已入库。

## 关键要点

- MD5去重是知识库质量保障的第一道防线，通过内容哈希比对避免相同文档重复入库，节省存储空间和检索噪声。
- `RecursiveCharacterTextSplitter`按分隔符优先级递归分割，优先在段落、句子等语义边界处切分，保证分块的语义完整性。
- 条件分割策略（长度阈值判断）避免对短文本进行不必要的分割，减少碎片化。
- 元数据（metadata）为每个文本块附加来源、时间等信息，检索时可用于结果溯源和过滤。
- `add_texts`方法内部自动调用嵌入模型进行向量化，开发者无需手动处理向量转换。

## 小结

本文详细介绍了`KnowledgeBaseService`的文档入库流程：MD5去重确保不重复入库，条件分割根据文本长度决定是否切分，元数据附加实现来源追溯，最终通过`Chroma.add_texts`完成向量化和存储。这个流程是RAG系统数据基础建设的核心环节，入库质量直接影响后续检索和生成的效果。
