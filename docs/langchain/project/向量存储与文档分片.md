# 向量存储与文档分片：构建知识库的基石

## 为什么需要向量存储？

大模型无法直接理解文本文件，RAG系统需要将文档转化为**向量（Embedding）**——一种高维数值表示，使得语义相似的文本在向量空间中距离更近。向量存储就是专门用来存储和检索这些向量的数据库。

## 核心概念

### 文档分片（Chunking）

原始文档通常很长，直接存入向量库会导致：
- 检索精度低（长文档包含多个主题，匹配不精准）
- 超出模型上下文窗口
- 向量表示不够聚焦

因此需要将长文档切分为合适大小的片段（chunk）。

### 向量化（Embedding）

将文本片段通过嵌入模型转换为向量，使得：
- 语义相似的文本向量距离近
- 可以通过向量相似度搜索找到相关文档

## 项目中的实现

### VectorStoreService类

```python
from langchain_chroma import Chroma
from langchain_core.documents import Document
from model.factory import embed_model
from langchain_text_splitters import RecursiveCharacterTextSplitter


class VectorStoreService:
    def __init__(self):
        # 初始化Chroma向量库
        self.vector_store = Chroma(
            collection_name=chroma_conf["collection_name"],   # 集合名称
            embedding_function=embed_model,                    # 嵌入模型
            persist_directory=chroma_conf["persist_directory"], # 持久化目录
        )

        # 初始化文本分片器
        self.spliter = RecursiveCharacterTextSplitter(
            chunk_size=chroma_conf["chunk_size"],       # 分片大小
            chunk_overlap=chroma_conf["chunk_overlap"],  # 重叠大小
            separators=chroma_conf["separators"],        # 分隔符
            length_function=len,                         # 长度计算函数
        )
```

### 配置参数

```yaml
# chroma.yml
collection_name: agent          # 向量库集合名称
persist_directory: chroma_db    # 持久化存储目录
k: 3                            # 检索返回的文档数量
data_path: data                 # 知识文件目录
md5_hex_store: md5.text         # MD5去重记录文件
allow_knowledge_file_type: ["txt", "pdf"]  # 允许的文件类型

chunk_size: 200                 # 每个分片最大200字符
chunk_overlap: 20               # 相邻分片重叠20字符
separators: ["\n\n", "\n", ".", "!", "?", "。", "！", "？", " ", ""]  # 分隔符优先级
```

## 递归字符分片器

`RecursiveCharacterTextSplitter`是LangChain中最常用的分片器，它的工作原理是：

1. 按照分隔符优先级依次尝试切分
2. 先用最大的分隔符（如双换行`\n\n`，即段落）
3. 如果切出的片段仍然超过`chunk_size`，则用下一个分隔符（如单换行`\n`）
4. 依此类推，直到片段大小合适

```
原始文档
  ↓ 按\n\n切分
段落1（< 200字符）✓
段落2（> 200字符）→ 按\n切分 → 句子1 ✓ + 句子2 ✓
段落3（< 200字符）✓
```

### chunk_overlap的作用

相邻分片之间保留20字符的重叠，确保：
- 跨分片的语义不会断裂
- 检索时不会遗漏边界处的重要信息

```
分片1: [......重叠区]
分片2:       [重叠区......重叠区]
分片3:                   [重叠区......]
```

## 文档加载与去重

### 加载流程

```python
def load_document(self):
    # 1. 获取所有允许类型的文件
    allowed_files_path = listdir_with_allowed_type(
        get_abs_path(chroma_conf["data_path"]),
        tuple(chroma_conf["allow_knowledge_file_type"]),
    )

    for path in allowed_files_path:
        # 2. 计算文件MD5
        md5_hex = get_file_md5_hex(path)

        # 3. 检查是否已处理过
        if check_md5_hex(md5_hex):
            logger.info(f"[加载知识库]{path}内容已经存在知识库内，跳过")
            continue

        # 4. 加载文档
        documents = get_file_documents(path)  # PDF或TXT

        # 5. 分片
        split_document = self.spliter.split_documents(documents)

        # 6. 存入向量库
        self.vector_store.add_documents(split_document)

        # 7. 记录MD5，避免重复加载
        save_md5_hex(md5_hex)
```

### MD5去重机制

每次加载文件前计算其MD5哈希值，与已记录的MD5列表比对：
- 已存在 → 跳过，避免重复入库
- 不存在 → 加载处理，完成后记录MD5

这种增量加载机制确保知识库可以持续更新而不会产生重复数据。

## 检索器

```python
def get_retriever(self):
    return self.vector_store.as_retriever(search_kwargs={"k": chroma_conf["k"]})
```

检索器将用户问题转为向量，在Chroma中搜索最相似的k个文档片段（本项目k=3）。

## 分片参数调优建议

| 参数 | 过小 | 过大 |
|------|------|------|
| chunk_size | 语义不完整，检索精度低 | 包含过多主题，匹配不精准，超出上下文 |
| chunk_overlap | 跨片信息丢失 | 重复内容过多，浪费存储和上下文空间 |

一般建议：
- `chunk_size`：200-1000字符，根据文档类型调整
- `chunk_overlap`：chunk_size的10%左右

## 总结

向量存储与文档分片是RAG系统的基础设施。合理的分片策略直接影响检索质量和最终回答的准确性。本项目使用Chroma作为向量数据库，配合递归字符分片器和MD5去重机制，构建了一个可增量更新的知识库系统，为智能客服提供专业的知识检索能力。
