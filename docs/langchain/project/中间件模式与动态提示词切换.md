# 中间件模式与动态提示词切换：让智能体更灵活

## 什么是中间件？

中间件（Middleware）是一种在软件系统中广泛使用的设计模式，它允许在不修改核心逻辑的前提下，在请求处理流程的特定位置插入自定义逻辑。在智能体系统中，中间件可以在**工具调用前后**和**模型推理前后**插入额外处理。

本项目使用了LangGraph提供的中间件机制，实现了三种中间件：

1. **工具调用监控**（`monitor_tool`）—— 包裹工具调用
2. **模型调用前日志**（`log_before_model`）—— 模型推理前
3. **动态提示词切换**（`report_prompt_switch`）—— 提示词生成前

## 三种中间件类型详解

### 1. wrap_tool_call —— 工具调用包裹

`@wrap_tool_call`装饰器包裹整个工具调用过程，可以在工具执行前后插入自定义逻辑：

```python
from langchain.agents.middleware import wrap_tool_call
from langchain.tools.tool_node import ToolCallRequest
from langchain_core.messages import ToolMessage

@wrap_tool_call
def monitor_tool(
        request: ToolCallRequest,       # 请求的数据封装
        handler: Callable[[ToolCallRequest], ToolMessage | Command],  # 工具执行函数
) -> ToolMessage | Command:
    # 工具调用前：记录日志
    logger.info(f"[tool monitor]执行工具：{request.tool_call['name']}")
    logger.info(f"[tool monitor]传入参数：{request.tool_call['args']}")

    try:
        # 执行工具
        result = handler(request)
        logger.info(f"[tool monitor]工具{request.tool_call['name']}调用成功")

        # 特殊处理：fill_context_for_report调用后修改运行时上下文
        if request.tool_call['name'] == "fill_context_for_report":
            request.runtime.context["report"] = True

        return result
    except Exception as e:
        logger.error(f"工具{request.tool_call['name']}调用失败，原因：{str(e)}")
        raise e
```

**核心功能**：
- 记录每次工具调用的名称和参数
- 捕获工具调用异常并记录
- **关键逻辑**：当`fill_context_for_report`工具被调用时，将运行时上下文中的`report`标记设为`True`

### 2. before_model —— 模型推理前

`@before_model`装饰器在每次模型推理前执行：

```python
from langchain.agents.middleware import before_model
from langchain.agents import AgentState
from langgraph.runtime import Runtime

@before_model
def log_before_model(
        state: AgentState,      # Agent的完整状态
        runtime: Runtime,       # 运行时上下文
):
    logger.info(f"[log_before_model]即将调用模型，带有{len(state['messages'])}条消息。")
    logger.debug(f"[log_before_model]{type(state['messages'][-1]).__name__} | {state['messages'][-1].content.strip()}")
    return None
```

**核心功能**：
- 记录模型调用前的消息数量
- 调试模式下记录最后一条消息的类型和内容
- 返回`None`表示不修改状态，继续正常流程

### 3. dynamic_prompt —— 动态提示词切换

`@dynamic_prompt`是最关键的中间件，它在每次生成提示词之前执行，允许根据运行时上下文动态切换提示词：

```python
from langchain.agents.middleware import dynamic_prompt, ModelRequest

@dynamic_prompt
def report_prompt_switch(request: ModelRequest):
    is_report = request.runtime.context.get("report", False)
    if is_report:               # 报告生成场景 → 使用报告提示词
        return load_report_prompts()

    return load_system_prompts()  # 普通场景 → 使用默认提示词
```

## 动态提示词切换的完整流程

这是项目中最精妙的设计，让我们用一个完整的流程来理解：

```
用户："给我生成我的使用报告"
  ↓
模型思考：这是报告生成需求 → 调用fill_context_for_report
  ↓
[wrap_tool_call中间件] 捕获到fill_context_for_report调用
  ↓
[wrap_tool_call中间件] 将runtime.context["report"]设为True
  ↓
模型继续思考：需要获取用户ID和月份 → 调用get_user_id, get_current_month
  ↓
[before_model中间件] 记录日志
  ↓
[dynamic_prompt中间件] 检测到context["report"]=True → 加载报告提示词
  ↓
模型使用报告提示词生成专业报告
```

### 初始状态

在`ReactAgent.execute_stream`中，初始上下文设置为：

```python
context={"report": False}  # 默认不是报告场景
```

### 触发切换

只有当智能体主动调用`fill_context_for_report`工具时，中间件才会将`report`标记设为`True`，从而在下一轮模型推理时切换到报告提示词。

## 为什么需要动态提示词切换？

同一个智能体可能需要处理不同类型的任务，每种任务需要不同的提示词策略：

| 场景 | 提示词 | 特点 |
|------|--------|------|
| 智能客服 | main_prompt.txt | ReAct思考流程，7个工具可用 |
| 报告生成 | report_prompt.txt | 专业报告写手，聚焦数据整理和报告输出 |

如果不做切换，用客服提示词去生成报告，输出格式和内容质量都会大打折扣。

## 中间件的设计优势

1. **解耦**：工具逻辑与监控逻辑分离，互不影响
2. **可扩展**：新增中间件无需修改现有代码
3. **运行时可配置**：通过`context`传递运行时状态，实现动态行为
4. **透明性**：工具本身不需要知道中间件的存在

## 总结

中间件模式是智能体架构中不可或缺的一环。它让我们能够在不侵入核心业务逻辑的前提下，实现日志监控、异常处理、上下文注入和动态提示词切换等横切关注点。特别是动态提示词切换机制，让一个智能体能够灵活应对多种业务场景，是构建复杂Agent系统的关键技术。
